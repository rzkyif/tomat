A **tool** is one thing the agent can do beyond talking: search the web, fetch a
page, run a calculation. Tools are how the agent acts in the world rather than
only describing it. A tool always comes from a provider, and there are two kinds:
an **extension**, a bundle you install or remove as a unit, and an
[MCP server](/manual/tools/mcp), a separate program tomat connects to. A fresh
Core comes with a built-in extension already installed, so the abilities are there
to switch on, with room to add more.

## How tools reach the agent

You rarely hand-pick tools for a message. Before each one, tomat ranks your
installed tools by how relevant they are to what you just asked, and offers the
agent only the best matches. A model handed every tool you own would waste its
attention sorting through them; a short, relevant list keeps it focused on the
task.

<RelevantToolsDemo />

A few tools you rely on regardless can be marked as always available, so they are
offered every time instead of waiting to be ranked in. The result, the ranked
shortlist plus your always-on tools, is what the conversation shows you for each
turn, as covered in [Inline Results](/manual/conversations/inline-results).

## Tools and extensions in Settings

tomat keeps the two concerns apart, because they are different jobs. The **Tools**
group is the single place to manage every tool from every provider, extension or
MCP server alike: turn each one on or off and set its permissions, all in one
list. Tool use is off until you turn it on there, so a fresh install talks but
does not act until you let it. The same group holds the dials for the selection
above: how many tools tomat offers per message, and which tools are always
available regardless of relevance.

<SettingsPanelDemo group="tools" />

The **Extensions** group is the other half: it is where you install, update, and
remove the packages themselves. Adding an extension is how you teach the agent new
abilities (and sometimes new [knowledge or skills](/manual/knowledge/memories),
since an extension can ship those too), and removing one takes them away cleanly,
since an extension is a self-contained bundle rather than a loose pile of tools.
The tools it brought then appear in the Tools group for you to switch on.

## Permissions and the sandbox

A tool can reach outside tomat, so each extension declares the permissions it needs
up front, such as access to a specific network host, and runs in a sandbox that
holds it to them. A tool can only do what you have allowed it to, and you can see
what it asked for before you let it run.

<Aside type="caution">
  A tool that reaches the network or runs commands acts on your behalf. Install extensions you
  trust, and read the permissions a extension declares before granting them.
</Aside>

Tools and your saved knowledge work the same way under the hood: tomat finds what
is relevant and offers it to the agent. Before the knowledge side, the next page
covers the other provider of tools: [MCP Servers](/manual/tools/mcp).
