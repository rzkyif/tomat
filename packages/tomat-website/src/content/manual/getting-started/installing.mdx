tomat comes in two parts: a **Client**, the app you look at, and a **Core**, the
service that does the work. You install the Client first, then point it at a Core.
The common setup runs both on your own machine and does everything locally, with
nothing sent anywhere until you decide otherwise, but the Core can just as well run
on a different computer (see [The Basics](/manual/getting-started/basics)).

<InstallDiagram />

They connect over the network, so one Core can serve many Clients and one Client
can reach many Cores. The picture above is one such spread: a laptop running both
halves, a phone that carries only the Client, and a desktop that runs a Core for the
heavier work. You do not have to plan any of that up front, though. Install the
Client, let it set up a Core on the same machine, and you have a complete local
setup; you can add or move Cores whenever you like (see
[Cores and Pairing](/manual/maintenance/cores)).

## Install the Client

Head to the [Install page](/install), pick your operating system, and download the
installer. Open it and follow the prompts, the same as any other app: a `.dmg` on
macOS and a setup installer on Windows. On Linux the Client is an **AppImage**, a
single portable file you make executable and run, and the build that keeps itself
up to date. Either way you get the **Client** alone, and you connect it to a Core
in the next step.

Prefer the command line, or installing over SSH onto a server with no desktop? The
same page has a one-line command next to the download buttons, and its **How to
Install** list walks each path step by step. The command runs the script published
at `get.au.tomat.ing/install`, so you can read it before you run it. Either way,
everything tomat places lives under your home directory; there is no system-wide
service to manage.

<Aside type="note">
  macOS blocks the downloaded app the first time you open it, since tomat is not yet signed by an
  identified Apple developer (that signing is on the way). Open System Settings, go to Privacy &
  Security, and choose **Open Anyway** next to the tomat message. The install command clears this
  for you, so it only affects the downloaded installer.
</Aside>

## Set up a Core

The first time you launch the Client it asks where its Core should run, since it
needs one to do anything. You have two choices:

- **On this computer.** The Client installs and starts a local Core for you and
  pairs with it automatically. This is the path most people want: everything stays
  on your machine, fully local.
- **On another computer.** Pair with a Core running elsewhere, such as a more
  powerful desktop. Install the Core there first, then pair the Client with it.
  See [Cores and Pairing](/manual/maintenance/cores).

Either way, once a Core is paired the Client remembers it and reconnects on its
own. You can pair more Cores or switch between them later under **Cores** in
Settings (see [Cores and Pairing](/manual/maintenance/cores)).

## What gets installed

A local setup leaves you with two things:

- The **Client**, the desktop app you open from Applications, your app menu, or
  the Start menu.
- A **Core**, with all of its state kept under `~/.tomat`.

That `~/.tomat` directory is where the Core keeps your sessions, memories, and
settings, along with any models you download later. Keeping it on disk is what
makes tomat local-first: your conversations are files on the machine running the
Core, not rows in someone else's database. (Snippets and the look of the app are
kept by the Client instead, so they travel with it; see
[The Settings Panel](/manual/maintenance/settings-panel).) If you ever want to take
it all back off, see [Uninstalling tomat](/manual/maintenance/uninstalling).

<Aside type="caution">
  Nothing leaves your machine until you point a setting at an **External** provider. When you do,
  the messages in those sessions are sent to that provider to be answered. See [Language
  Model](/manual/models/language-model).
</Aside>

## Staying current

tomat keeps itself up to date. When a new release is available, the app offers to
install it, and the Core can stage and apply its own update in the background,
rolling back if anything goes wrong. You can also download the latest installer
(or re-run the install command) at any time to jump straight to the newest build.

How new those builds are is set by your **channel**, which you can leave on its
default for now. When you want to try faster-moving builds without disturbing
your setup, see [Updates and Channels](/manual/maintenance/updates-and-channels).
